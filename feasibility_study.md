# Feasibility Study: Public GitHub Repository Flow & Value Stream Analytics

This document evaluates the feasibility of building a flow metrics dashboard using public GitHub repository data. The goal is to demonstrate the relationship between AI coding assistance, moving constraints (Theory of Constraints), and high-level outcomes mapped to the Scrum.org Evidence-Based Management (EBM) framework.

---

## Executive Summary

Yes, it is highly feasible to access public GitHub repository data to reconstruct developer flow metrics, identify moving bottlenecks, detect AI tool usage, and approximate Scrum.org EBM metrics.

While public repositories do not expose private business data (such as proprietary financial metrics, internal Jira sprint configurations, or developer salary details), the GitHub REST and GraphQL APIs expose granular event histories. These histories allow us to construct high-fidelity proxies for process efficiency (Time-to-Market), quality and capability (Ability to Innovate), and user traction (Value).

By correlating explicit and heuristic AI signatures with downstream queue times, we can show whether AI tools are genuinely accelerating organizational velocity or merely shifting the system bottleneck downstream and creating unreleased inventory.

---

## 1. Metric Framework Mapping (Yuval's POV)

The core thesis is that **AI coding tools locally optimize the coding phase**. If coding is not the system's current bottleneck, this acceleration does not improve overall delivery outcomes. Instead, it increases inventory (open pull requests, unreleased code), which increases downstream review times, context switching, and regression bugs, ultimately degrading the organization's strategic agility.

To prove or disprove this, the dashboard must capture metrics across three distinct layers:

```
[ AI Usage Indicators ]  ==>  [ Downstream Bottlenecks (TOC) ]  ==>  [ EBM Strategic Outcomes ]
- Co-author git trailers        - Open PR queue size (WIP)           - Time-to-Market (T2M)
- Bot actor accounts           - Review wait times                  - Ability to Innovate (A2I)
- Comment signatures           - Merge delays                       - Current & Unrealized Value
- High-density commit bursts   - Release cycle latency
```

### Layer A: AI Usage Detection
Because inline autocomplete (like GitHub Copilot in the IDE) is "silent" on the Git commit level, we must combine explicit metadata signatures with heuristic patterns:

1. **Explicit Metadata Signatures (Commit level)**:
   - **GraphQL Git Trailers**: The GitHub GraphQL API automatically parses the `Co-authored-by` trailer in commits. We can scan for:
     - `Co-authored-by: GitHub Copilot <noreply@github.com>` or `Co-authored-by: Copilot <copilot@github.com>`
     - `Co-authored-by: aider <aider@aider.chat>` or references to `aider (<model>)`
     - Claude Code signatures (e.g., `Co-authored-by: Claude <claude@anthropic.com>`)
2. **Explicit Bot Activity (PR/Issue/Comment level)**:
   - Contributions made by registered AI application bots (e.g., `github-copilot[bot]`, `coderabbit[bot]`, `sweep-ai[bot]`, `devin-ai[bot]`, `dependabot[bot]`).
3. **PR Description and Comment Text Signatures**:
   - Automated reviews or descriptions containing text footprints (e.g., `*Generated by Coderabbit*`, `*Summary by Claude*`, `*Refactored by Sweep*`).
4. **Heuristic Patterns (Heuristic AI Footprint)**:
   - **Commit Density / Typing Speeds**: Non-trivial commits submitted by the same user within minutes or seconds of each other.
   - **High-Churn Commits**: Large additions and deletions that frequently modify the same file blocks, suggesting recursive AI regeneration rather than surgical human edits.

### Layer B: Process Efficiency & Downstream Bottlenecks (Theory of Constraints)
To detect bottleneck shifts, we split the lifecycle of a contribution into sequential stages:

1. **Coding Phase**: Time from the first commit of a branch/PR to the creation of the Pull Request.
2. **Review/QA Phase**: Time from Pull Request creation to the first review, and then to approval.
3. **Integration Phase**: Time from Pull Request approval to the merge event.
4. **Release Phase**: Time from Pull Request merge to the inclusion of the merge commit in a formal GitHub Release or Tag.

**Theory of Constraints Indication**: If AI adoption drops the *Coding Phase* duration, but *Review/QA Phase* or *Integration Phase* times spike, the bottleneck has shifted downstream. The organization is building up unmerged inventory, which increases complexity and review overhead.

### Layer C: Scrum.org Evidence-Based Management (EBM)
Using public repository metadata, we can construct the following EBM metrics:

| EBM Key Value Area (KVA) | EBM Metric Definition | Public GitHub API Proxy |
| :--- | :--- | :--- |
| **Time-to-Market (T2M)** | **Cycle Time** | Average duration from Issue assignment (or first PR commit) to PR merge. |
| | **Release Frequency** | Average number of GitHub Releases or Git tags created per month. |
| **Ability to Innovate (A2I)** | **Defect Ratio** | Ratio of closed issues labeled `bug` to those labeled `feature` or `enhancement`. |
| | **Technical Debt Ratio** | Ratio of Dependabot/refactoring PRs to feature PRs. |
| | **Context Switching** | Average number of open PRs/branches per active contributor. |
| | **Code Churn Index** | Ratio of lines deleted/modified within 14 days of being written. |
| **Current / Unrealized Value**| **User Engagement** | Stars, forks, watches, external issue creation frequency, and external comment activity. |

## 2. GitHub API Feasibility & Data Points

The following table summarizes the feasibility of extracting these metrics using public GitHub APIs (REST and GraphQL):

| Data Area | Required Metrics | Feasible? | API Endpoint / GraphQL Fields | Notes & Limitations |
| :--- | :--- | :--- | :--- | :--- |
| **PR Lifecycle** | Cycle times, review history, approval states | **Yes** | GraphQL `PullRequest` -> `timelineItems` (filter: `PullRequestReview`, `MergedEvent`, `ClosedEvent`) | Highly granular. Captures exact timestamps of creation, review requests, approvals, and merges. |
| **Commit History** | Code churn, co-authors, typing speeds | **Yes** | GraphQL `Commit` -> `history` -> `authors`, `additions`, `deletions` | `authors` node automatically parses `Co-authored-by` Git trailers for AI tools. |
| **Issues / Backlog** | Lead times, bug ratios, prioritization | **Yes** | GraphQL `Repository` -> `issues` -> `timelineItems`, `labels` | Tracks creation to closure times. Limited by how consistently a public repo uses GitHub labels. |
| **Releases / Tags** | Release frequency, deployment mapping | **Yes** | GraphQL `Repository` -> `releases` or `refs(refPrefix: "refs/tags/")` | Provides exact deployment/release markers. |
| **Bot Identification**| Automated PRs, reviews, descriptions | **Yes** | GraphQL `Actor` -> `login` (checking for `[bot]` suffix or specific logins) | 100% reliable for registered GitHub apps. |
| **Work Allocation** | Feature vs. Debt vs. Overhead | **Partial** | GraphQL `Repository` -> `issues`/`pullRequests` -> `labels` | Relies entirely on the repo's labeling hygiene (e.g. `type: feature`, `type: bug`, `type: tech-debt`). |

### Recommended GraphQL Query Structure
To minimize payload size and avoid API rate limits, the GraphQL API is preferred over the REST API. Below is a sample query to fetch PR lifecycle metrics and AI co-author signatures:

```graphql
query GetPRFlowData($owner: String!, $name: String!, $cursor: String) {
  repository(owner: $owner, name: $name) {
    pullRequests(first: 50, after: $cursor, states: [MERGED, CLOSED], orderBy: {field: CREATED_AT, direction: DESC}) {
      pageInfo {
        hasNextPage
        endCursor
      }
      nodes {
        number
        title
        createdAt
        closedAt
        merged
        mergedAt
        additions
        deletions
        author {
          login
          __typename
        }
        commits(first: 100) {
          totalCount
          nodes {
            commit {
              committedDate
              additions
              deletions
              authors(first: 5) {
                nodes {
                  name
                  email
                  user {
                    login
                  }
                }
              }
            }
          }
        }
        reviews(first: 50) {
          nodes {
            createdAt
            state
            author {
              login
              __typename
            }
          }
        }
      }
    }
  }
}
```

## 3. Implementation Challenges & Mitigations

### Challenge 1: Public Repo Labeling Hygiene
* **Problem**: Many public repos do not use structured labels (e.g., distinguishing bugs from features), making EBM Ability to Innovate (A2I) calculations difficult.
* **Mitigation**: Use simple natural language processing (NLP) heuristics on issue/PR titles and descriptions. If a title contains "fix", "bug", "crash", or "broken", classify it as a defect. If it contains "feat", "add", "new", or "implement", classify it as a feature.

### Challenge 2: "Silent" AI Autocomplete Usage
* **Problem**: Standard IDE autocompletes (GitHub Copilot without agentic commits) leave no Git trailer.
* **Mitigation**: Use **Commit Density** and **Typing Speed** metrics. If a contributor commits 4 separate files with 200+ lines of changes and writes a detailed commit description within 90 seconds, it is marked as AI-assisted. In addition, monitor PRs that integrate AI-powered review assistants (like Coderabbit) which leave comment signatures.

### Challenge 3: GitHub API Rate Limits
* **Problem**: The GitHub API limits unauthorized requests to 60/hour, and authorized requests (using a personal access token) to 5,000/hour.
* **Mitigation**: Since we are analyzing public repos, we can run the collection process asynchronously as a CLI script, cache the raw JSON payloads locally or in a SQLite database, and build the dashboard interface around this cached static data.

## 4. Proposed Dashboard Architecture

To align with Yuval's preferences for pragmatic, file-backed, and highly responsive systems, we propose a lightweight, decoupled stack:

```
[ Public GitHub API ]
       │  (GitHub GraphQL API)
       ▼
[ CLI Collector (Python / Node) ] ──> Stores static JSON files in [ /data ]
       │
       ▼
[ Static Dashboard (Astro / Svelte / Tailwind) ] ──> Hosted on Cloudflare Pages
```

1. **The Collector**: A simple CLI tool written in Node.js or Python. It takes a public repository URL, queries the GraphQL API, processes the raw event streams, computes the flow and EBM metrics, and saves them as a structured JSON file in a local data folder.
2. **The Cache**: A local `/data` directory containing repository-specific JSON files. This ensures the dashboard loads instantly and avoids running into GitHub API rate limits.
3. **The Dashboard**: A responsive, modern single-page dashboard built using Astro or Vite. It renders:
   - **The Lead Time Breakdown**: Visualizing Coding Time vs. Review Time vs. Merge Time.
   - **The WIP Inventory Curve**: Showing the correlation between AI commit density and the build-up of open PRs.
   - **The EBM Scorecard**: Rating the repository's Time-to-Market (T2M) and Ability to Innovate (A2I) over time.
   - **The Bottleneck Tracker**: Pointing out where the system constraint lies at any given week (e.g., "Review Queue" or "Release Latency").
